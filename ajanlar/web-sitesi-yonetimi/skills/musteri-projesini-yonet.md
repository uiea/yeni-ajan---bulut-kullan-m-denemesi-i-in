# Skill: Manage a Client Project

## Purpose
Route a web-site request to the correct confirmed client and project, then organize the resulting local work safely.

## When to Use This Skill
- "[Firma] sitesini güncelle"
- "[Firma] projesindeki dosyalara bak"
- "Bu müşterinin landing page revizyonunu hazırla"
- "[Proje adı] teslim durumunu düzenle"

## Do NOT Use This Skill For
- Adding a brand-new client -> use `yeni-musteri-ekle.md`.
- Creating a new project folder -> use `yeni-proje-baslat.md`.

## Serves Goal
- Route work to the right client.
- Preserve safe delivery control.

## Inputs
- `musteriler.md`.
- The matching client skill.
- The matching project folder and documentation.
- The member's requested task and any current approval.

## Process
1. Identify the client from the registry. If the name, domain, or project is ambiguous, ask the member to choose before reading or saving client-specific work.
2. Read the client skill and project brief, then restate the active client and project in the working note.
3. Check `teslimler.md` and `erisimler.md` for confirmed status and references. Never reveal secret values.
4. Make local drafts or organize local files only inside the selected project folder.
5. Record material decisions, revisions, and handoff readiness in the project documentation.
6. Before any external transmission, deployment, upload, live-site change, or client message, show the exact destination and final content, then wait for action-specific approval.

## Output
- A correctly scoped local draft, project update, or status note.
- Updated local project documentation when facts are confirmed.

## Quality Bar
- The response identifies one client and one project before client-specific work starts.
- All files stay inside the selected project's folder.
- Any external or live-site action remains pending until explicit approval.

## Rules & Feedback
- Treat the registry and client skill as the routing source of truth.
  Why: client names alone are not always unique or complete.
  How to apply: read both before acting on a known client request.
- Do not infer approval from a request to prepare work.
  Why: preparing a draft and publishing a change are different actions.
  How to apply: draft locally first; request a specific confirmation for external delivery.
